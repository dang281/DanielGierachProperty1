# Technical SEO Audit, danielgierach.com

Date: 2026-05-10
Scope: crawlability, indexing, site architecture, sitemaps, robots, redirects, canonicals, structured data, URL structure.
Source of truth: source files under `/src` and built output under `/dist` (generated by `astro build`).

## Summary

Indexing and site architecture are mostly healthy: every page renders a canonical, an articulated global JSON-LD graph (RealEstateAgent + Person + WebSite + identity FAQ) is injected on every Layout-served page, breadcrumbs are emitted as JSON-LD, robots.txt is present, and the build is fully static. There are, however, several material issues that are quietly leaking equity or creating crawl-budget waste. Two sitemaps are being published in parallel and they disagree by 466+ URLs, with the custom sitemap missing all 27 resource pages, the Newstead suburb, the `water-sewer-map` tool, and roughly 435 of the 573 insights articles. The Astro-generated sitemap goes the other way and includes pages that should be excluded (`/dashboard/`, `/lp/*`, `/social-preview/`). Three pages that should be `noindex` are currently being served as `index, follow`: `/thank-you`, `/404`, and `/privacy`/`/terms` are debatable (not technically wrong but generally low quality and worth excluding from crawl). Two internal pages (`/dashboard`, `/social-preview`) attempt a JS-based `noindex` that Google often does not honour. The custom sitemap and canonicals also disagree on trailing slashes, and tool pages all carry the title suffix "Dan's Website" rather than the site brand. Nothing is catastrophic. Everything below is fixable in a few small edits.

## Detail per finding

### TSEO-001, dual sitemap conflict (P0)

Evidence:
- `public/robots.txt` lines 8-9 declares two sitemaps:
  - `Sitemap: https://danielgierach.com/sitemap-index.xml`
  - `Sitemap: https://danielgierach.com/sitemap.xml`
- `astro.config.mjs` line 10 enables `@astrojs/sitemap`, which generates `dist/sitemap-index.xml` pointing to `dist/sitemap-0.xml`.
- `src/pages/sitemap.xml.ts` is a custom 300-line route that emits `dist/sitemap.xml`.
- The two sitemaps disagree:
  - `sitemap-0.xml` contains 721 URLs (every built page including disallowed `/dashboard/`, `/lp/*`, `/social-preview/`).
  - `sitemap.xml` contains 255 URLs (a hand-curated subset).

What is wrong: Google receives two sitemaps with overlapping but conflicting URL sets, with different lastmod values (Astro's are absent or per-build, custom is `today`), different trailing slash conventions, and one of the sitemaps contains pages that the same robots.txt disallows. This produces "Indexed, though blocked by robots.txt" warnings and sends mixed signals about which canonical version of each URL to prefer.

Why it matters: Search engines respect both sitemaps. URLs in only one can still be discovered, but the inconsistency wastes crawl budget and creates avoidable Search Console errors. With 716+ pages on a brand-new site, every cleanly-included URL increases the chance of timely indexing.

Recommended fix: Pick one. Either:
1. Remove the custom `sitemap.xml.ts`, drop the second `Sitemap:` line in robots.txt, and add the missing exclusions (`/dashboard`, `/lp`, `/social-preview`, `/preview`, `/thank-you`, `/privacy`, `/terms`, `/404`) to the Astro `sitemap()` filter.
2. Or remove the Astro sitemap integration, drop `Sitemap: https://danielgierach.com/sitemap-index.xml` from robots.txt, and update `sitemap.xml.ts` to enumerate every published URL by reading the filesystem (or by importing the suburb/tool/insight/resource lists used by `getStaticPaths`).

Option 1 is lower risk and lower maintenance.

### TSEO-002, indexable thank-you, 404, privacy, terms (P0)

Evidence (from built HTML):
- `dist/thank-you/index.html`: `<meta name="robots" content="index, follow, max-snippet:-1, max-image-preview:large, max-video-preview:-1">` and `<link rel="canonical" href="https://danielgierach.com/thank-you/">`.
- `dist/404.html`: same robots and `<link rel="canonical" href="https://danielgierach.com/404/">`.
- `dist/privacy/index.html`: same robots, canonical `/privacy/`.
- `dist/terms/index.html`: same robots, canonical `/terms/`.

What is wrong: The thank-you page is a thin "Thanks for getting in touch" page that should never rank in search. The 404 page is non-content. Both are emitted with `index, follow` because `Layout.astro` line 269 only sets `noindex, nofollow` when `draft` is true and Astro happens to render the 404 file as `dist/404.html` (Vercel will serve it on missing paths but it is also reachable directly at `/404/`).

Privacy and terms pages are typically `index, follow` is acceptable, but they are not high-quality landing pages and many sites set them `noindex` to focus crawl budget on revenue-driving pages. This is judgement call territory.

Why it matters: An indexable `/thank-you` is a known issue because conversion tracking referrals can leak the URL into Google. Once indexed, anyone searching the site or for a stray phrase from the thank-you page can land there directly, bypassing the form (and skewing GA4 conversion numbers). An indexable 404 is less harmful but creates a duplicate "this page does not exist" entry against the catch-all 404 served by Vercel.

Recommended fix:
- In `src/pages/thank-you.astro` and `src/pages/404.astro`, pass `draft={true}` to `<Layout>` (or add a dedicated `noindex` prop). The existing `draft` prop already wires into the robots meta on line 269.
- Optionally exclude `/privacy` and `/terms` from indexing too. Sitemap already excludes them via the Astro filter, so they will be hard to discover, but they remain `index, follow`.

### TSEO-003, Astro sitemap leaks disallowed pages (P1)

Evidence:
- `dist/sitemap-0.xml` contains entries for:
  - `https://danielgierach.com/dashboard/`
  - `https://danielgierach.com/lp/bulimba/`, `/lp/camp-hill/`, `/lp/hawthorne/`, `/lp/murarrie/`, `/lp/seven-hills/`
  - `https://danielgierach.com/social-preview/`
- `public/robots.txt` lines 3-5 disallow `/dashboard/`, `/lp/`, `/preview/`.
- `astro.config.mjs` lines 11-15 only filter `thank-you`, `privacy`, `terms`, `404`.

What is wrong: A sitemap should never list a disallowed URL. Google fetches the sitemap, follows the URLs, hits robots.txt, logs "submitted URL blocked by robots.txt" warnings, and the URLs end up in the Excluded report.

Why it matters: This is the most common cause of false-positive Search Console errors. It also tells Google that the site does not know which pages it wants indexed.

Recommended fix: Extend the filter in `astro.config.mjs`:

```js
sitemap({
  filter: (page) =>
    !page.includes('/thank-you') &&
    !page.includes('/privacy') &&
    !page.includes('/terms') &&
    !page.includes('/404') &&
    !page.includes('/dashboard') &&
    !page.includes('/social-preview') &&
    !page.includes('/lp/'),
}),
```

The `/preview/` files are static assets in `public/preview/` and do not appear in the Astro sitemap (Astro only lists Astro-rendered pages), so no filter needed there.

### TSEO-004, custom sitemap is missing 466+ URLs (P1)

Evidence:
- `src/pages/sitemap.xml.ts` produces 255 URLs.
- The build produces 47 suburb pages; sitemap lists 46 (missing `newstead`).
- The build produces 56 tool pages; sitemap lists 55 (missing `water-sewer-map`).
- The build produces 26 resource pages; sitemap lists 0 (`grep '/resources/' src/pages/sitemap.xml.ts` returns nothing).
- The build produces 573 insights pages; sitemap lists ~138 explicitly (the rest are unreferenced).

What is wrong: The custom sitemap is a hand-maintained allowlist that has fallen behind. New articles, tools, and the entire `/resources` section never made it in.

Why it matters: If the custom sitemap is the one Google chooses to trust (it has lastmod values, the Astro one does not), 466+ URLs are effectively orphaned from sitemap-driven discovery. They will still be crawled via internal links, but slowly.

Recommended fix: Tied to TSEO-001. If keeping the custom sitemap, replace the hand-coded array with an enumeration of files under `src/pages/` (or import the same data sources used by `getStaticPaths` for resources, suburbs, tools, insights). Easiest and lowest risk: delete the custom sitemap and rely on the Astro-generated one with a tightened filter.

### TSEO-005, JS-based noindex on dashboard and social-preview (P1)

Evidence:
- `src/pages/dashboard.astro` line 3: `<script>document.querySelector('meta[name="robots"]')?.setAttribute('content','noindex,nofollow');</script>`
- `src/pages/social-preview.astro`: identical pattern.
- The initial HTML response on both pages includes `<meta name="robots" content="index, follow, ...">`, set by `Layout.astro` line 269.

What is wrong: Googlebot can render JavaScript, but the initial HTML signal is `index, follow`. Many bots and many crawl events use the static HTML response; rendered DOM mutations are not always honoured. Worse, Bing and most non-Google crawlers do not run JS at all.

Why it matters: Both pages are in the Astro sitemap (TSEO-003), are served `index, follow` in the initial HTML, and rely on a JS hack to flip that. The dashboard is password-gated but the index signal is still being broadcast. Social-preview is internal-only.

Recommended fix: Add a `noindex` prop (or reuse `draft={true}`) to `<Layout>` and remove the JS override. Apply to both `dashboard.astro` and `social-preview.astro`.

### TSEO-006, trailing slash inconsistency (P1)

Evidence:
- `dist/suburbs/seven-hills/index.html` canonical is `https://danielgierach.com/suburbs/seven-hills/`.
- The custom sitemap (`src/pages/sitemap.xml.ts`) lists `/suburbs/seven-hills` (no trailing slash) for the same page.
- Suburb page-level JSON-LD url fields are also no-trailing-slash: `"url":"https://danielgierach.com/suburbs/seven-hills"` (`src/pages/suburbs/seven-hills.astro` line 19).
- Hub pages: `src/pages/insights/contracts-and-settlement.astro` line 9: `"url": "https://danielgierach.com/insights/contracts-and-settlement"` but built canonical is `/insights/contracts-and-settlement/`.
- Astro sitemap uses trailing slashes throughout.

What is wrong: Three different sources give Google three different "canonical" representations of the same page. Astro's default `trailingSlash: 'ignore'` plus Vercel's default behaviour produces the trailing-slash variant on the canonical, but the team-maintained pieces (custom sitemap, hand-coded JSON-LD `url` fields) use the no-slash variant.

Why it matters: This is exactly the failure mode that produces "Duplicate, Google chose different canonical than user" in Search Console. It is also the most common reason a site with strong content underperforms in indexing.

Recommended fix: Pick one. The site is already publishing trailing-slash canonicals, so the simplest fix is to make every other URL match. Concretely:
1. Add `trailingSlash: 'always'` to `astro.config.mjs` (this also normalizes incoming requests in dev/build).
2. Update all `path:` entries in `sitemap.xml.ts` to include a trailing slash.
3. Update every page-level JSON-LD `url` to include a trailing slash.
4. Keep the canonical URL helper in `Layout.astro` line 26: `new URL(Astro.url.pathname, 'https://danielgierach.com').href` will include the trailing slash if the source path has one.

### TSEO-007, all 55 tool pages titled "Dan's Website" (P1)

Evidence:
- `src/layouts/ToolLayout.astro` line 14: `<Layout title={`${title} | Dan's Website`} description={description}>`.
- Built output: `<title>Mortgage Calculator | Dan&#39;s Website</title>` in `dist/tools/mortgage/index.html` and 54 other tool pages.
- The rest of the site uses `Daniel Gierach Property` or `Daniel Gierach | Real Estate Agent Brisbane` as the brand suffix.

What is wrong: "Dan's Website" is informal placeholder text. SEO impact is small but real: brand consistency matters for SERP CTR, branded queries, and Google's understanding that all 716 pages belong to the same entity.

Why it matters: Search snippets show this title verbatim. It also undermines the otherwise carefully-built RealEstateAgent + Person + Organization graph, which says the publisher is "Daniel Gierach Property", not "Dan's Website".

Recommended fix: In `src/layouts/ToolLayout.astro` line 14, change to:

```astro
<Layout title={`${title} | Daniel Gierach Property`} description={description}>
```

Or push the title pattern up to a single source of truth in `Layout.astro` so all sub-layouts inherit it.

### TSEO-008, breadcrumb JSON-LD URL inconsistency (P2)

Evidence:
- `src/layouts/Layout.astro` line 343 builds breadcrumb item URLs as `https://danielgierach.com${crumb.href}`.
- The home crumb has `href: '/'`, producing `https://danielgierach.com/` (trailing slash).
- Subsequent crumbs use `_bPath` which strips trailing slashes (line 29: `Astro.url.pathname.replace(/\/$/, '')`), producing `https://danielgierach.com/insights` and `https://danielgierach.com/insights/contracts-and-settlement`.
- This contradicts the canonical, which has trailing slashes.

What is wrong: Same root cause as TSEO-006. The breadcrumb URLs published in JSON-LD are no-trailing-slash; the canonical is trailing-slash. Validators flag this.

Why it matters: Breadcrumbs render in Google SERPs. Inconsistent URLs reduce the chance Google trusts the breadcrumb data.

Recommended fix: Normalize. Change line 29 to either preserve trailing slashes or always add one. With TSEO-006 settled (trailing slash everywhere), update Layout.astro to use the same convention.

### TSEO-010, duplicate LocalBusiness/RealEstateAgent entities on suburb pages (P2)

Evidence:
- `src/pages/suburbs/seven-hills.astro` lines 13-37 emit a `["LocalBusiness", "RealEstateAgent"]` JSON-LD block with `name: "Daniel Gierach, Ray White The Collective"` and addressLocality of the suburb.
- `src/layouts/Layout.astro` lines 71-153 emit a global `["RealEstateAgent", "LocalBusiness"]` block with `@id: "https://danielgierach.com/#business"` and addressLocality of `Bulimba`.
- Suburb-level block has no `@id`, so validators see two separate businesses.

What is wrong: The page declares two independent business entities (one global, one per-suburb), with slightly different names ("Daniel Gierach" vs "Daniel Gierach, Ray White The Collective") and different addresses. This confuses graph consumers (Knowledge Graph, Bing, Yandex).

Why it matters: For local SEO, having one canonical Business entity with `areaServed` (already present in the global graph) is preferred over many micro-business entities. The current pattern is not invalid but creates redundancy.

Recommended fix: Either:
- Drop the per-suburb `LocalBusiness` block. The global graph already covers `areaServed`.
- Or keep it but add `@id: "https://danielgierach.com/#business"` so it merges with the global entity, and use the per-suburb data only as additional `areaServed` entries.

Note: This is schema-only. No visible page change.

### TSEO-011, Tawk.to widget loads with placeholder IDs (P2)

Evidence:
- `src/layouts/Layout.astro` line 582: `s1.src = 'https://embed.tawk.to/TAWK_PROPERTY_ID/TAWK_WIDGET_ID';`
- This is the literal string `TAWK_PROPERTY_ID/TAWK_WIDGET_ID`, not a real property ID.

What is wrong: Every page load attempts to load `https://embed.tawk.to/TAWK_PROPERTY_ID/TAWK_WIDGET_ID`, which returns a 4xx (verify with curl). This is wasted network and can show as a console error to crawlers running in headless browsers.

Why it matters: Pure performance/health hit. Not directly SEO, but Lighthouse and Web Vitals notice failed third-party requests. Also surfaces in DevTools every page load.

Recommended fix: Either remove the Tawk.to script entirely (the comment says it is unconfigured) or set the real IDs.

### TSEO-016, /listings sitemap priority and changefreq are inflated (P3)

### TSEO-019, hub page url field omits trailing slash (P2)

Evidence:
- `src/pages/insights/contracts-and-settlement.astro` line 9: `"url": "https://danielgierach.com/insights/contracts-and-settlement"`.
